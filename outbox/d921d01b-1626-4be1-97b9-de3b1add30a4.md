## git worktree をシンプルに扱う

git worktree を使い慣れるのはやや大変です（個人の感想です）。

-   worktree の配置どうしようかなぁ
    
-   引数は worktree の場所と branch だよな、あれ、順序は？
    
-   worktree への移動めんどくさい
    

ショートカットを自作してみたり丁度良いツールないだろうかと探してみたりと逡巡を経て私は一旦 **git-wt** に落ち着きました。これによって以下のメリットを享受できました。

-   worktree の配置・移動を気にしない（config で指定）
    
-   repository 内に worktree を作っても自動で ignore してくれる
    
-   worktree 作成時の hook があり開発環境並列化に役立ちそう
    

使い方を比較してみますと、以下のような感じになります。

```
# git worktree の場合
$ git worktree add ../feature-a -b feature-a  # branch が存在する場合は -b をつけない
$ cd ../feature-a

# git-wt の場合
$ git wt feature-a  # branch が存在しなければ作成して移動する
```

## DB 環境の分離

Coding Agent さんにお仕事をお願いする際のベストプラクティスとして"自己検証ループを回す"というのがあります。つまり、リンターやテストの実行をしてもらいます。  
テストが DB の状態に依存するのは、モックやスタブを使っていてもよくあることかと思います。そして次のような問題が出てきます。

-   同じ DB にアクセスしてデータが意図せぬ状態になりテストが失敗する
    
-   じゃあ別にコンテナを立てて分離しよう→ポートが衝突してしまう
    
-   しょうがないからテストを直列実行する
    

これら問題を回避するための方法をいくつか見ていきます。

### DBレベルでの分離

別の DB を隣に立ててしまう、というアプローチです。

新たに作成した DB であれば自由に変更できるため、既存環境へ影響を与えずに検証や作業を進められます。セットアップについては、git-wt の hook に DB 初期化スクリプトを仕込んでおくとよいでしょう。  
あわせて、接続クライアント側の設定も新しい DB を参照するよう書き換える必要があります。

コンテナを複数立ち上げる方法と比較すると、リソース消費が少なく、より軽量に運用できる点がメリットです。ただし、不要になったリソースのお掃除のことを考える必要があります。

### コンテナ分離: Compose ファイルの Merge

コンテナポート衝突問題の回避策その1です。

**compose.override.yaml** というファイルを作成すると compose.yaml と Merge されるので、これでポートを上書きします。。また、docker compose コマンドの -f オプションで Compose ファイルを複数指定した場合にも Merge されます。

以下サンプルコードです。

```
# compose.yaml
services:
  mysql:
    image: mysql:8
    platform: linux/amd64
    environment:
      MYSQL_ALLOW_EMPTY_PASSWORD: "yes"
    volumes:
      - mysql-data:/var/lib/mysql
    ports:
      - 3306:3306

volumes:
  mysql-data:
```

```
# compose.override.yaml
services:
  mysql:
    ports: !override
      - 13306:3306
```

ポイントは **!override** の部分です。デフォルトでは Merge されるのでこれがないと上書きではなくポートのマッピングが2つ並ぶ形になります。  
ただし、どのように Merge されるかは環境変数やボリュームなど種類によって異なるため詳しくはドキュメントをご一読ください。

### コンテナ分離: Testcontainers の使用

コンテナポート衝突問題の回避策その2です。

Testcontainers を用いてテスト実行時にコンテナを起動するというアプローチです。リソース消費はやや多くなりそうですが、テストを実行するために事前にコンテナを起動しておく必要がないのも嬉しい点です。

以下 Go でのサンプルコードです。

```
// calculate.go
package testcontainerssample

import (
	"context"
	"database/sql"
)

func CalculateSum(ctx context.Context, db *sql.DB, x, y int) (int, error) {
	var result int
	err := db.QueryRowContext(ctx, "SELECT ? + ?", x, y).Scan(&result)
	if err != nil {
		return 0, err
	}

	return result, nil
}
```

```
// calculate_test.go
package testcontainerssample

import (
	"context"
	"database/sql"
	"fmt"
	"testing"

	_ "github.com/go-sql-driver/mysql"
	"github.com/stretchr/testify/require"
	"github.com/testcontainers/testcontainers-go"
	"github.com/testcontainers/testcontainers-go/wait"
)

func TestCalculateSum_WithMySQL(t *testing.T) {
	t.Parallel()

	ctx := context.Background()
	db := openMySQLForTest(t, ctx)

	cases := []struct {
		name string
		x    int
		y    int
		want int
	}{
		{name: "positive numbers", x: 10, y: 32, want: 42},
		{name: "includes zero", x: 0, y: 7, want: 7},
		{name: "negative and positive", x: -3, y: 10, want: 7},
		{name: "both negative", x: -8, y: -4, want: -12},
	}

	for _, tc := range cases {
		t.Run(tc.name, func(t *testing.T) {
			got, err := CalculateSum(ctx, db, tc.x, tc.y)
			require.NoError(t, err)
			require.Equal(t, tc.want, got)
		})
	}
}

func openMySQLForTest(tb testing.TB, ctx context.Context) *sql.DB {
	tb.Helper()

	const (
		mysqlDatabase = "testdb"
	)

	container, err := testcontainers.Run(
		ctx,
		"mysql:8",
		testcontainers.WithExposedPorts("3306/tcp"),
		testcontainers.WithEnv(map[string]string{
			"MYSQL_ALLOW_EMPTY_PASSWORD": "yes",
			"MYSQL_DATABASE":             mysqlDatabase,
		}),
		testcontainers.WithWaitStrategy(
			wait.ForListeningPort("3306/tcp"),
			wait.ForLog("ready for connections"),
		),
	)
	testcontainers.CleanupContainer(tb, container)
	require.NoError(tb, err)

	endpoint, err := container.PortEndpoint(ctx, "3306/tcp", "")
	require.NoError(tb, err)

	dsn := fmt.Sprintf("root@tcp(%s)/%s?parseTime=true", endpoint, mysqlDatabase)
	db, err := sql.Open("mysql", dsn)
	require.NoError(tb, err)
	tb.Cleanup(func() {
		_ = db.Close()
	})
	require.NoError(tb, db.PingContext(ctx))

	return db
}
```

一見すると「これでホストポート衝突しないの？」と思うかもしれませんが container.PortEndpoint(…) が localhost:<mappedportfor-3306> を返してくれるので大丈夫です。

一応 testcontainers.CleanupContainer(…) を呼んでいますが reaper というコンテナが動いていて最終的には掃除してくれるようです。上記の書き方ですとテスト関数ごとにコンテナが立つことになるので都度掃除しておきたいです。

## Untracked files の管理

.env のような git で追跡されないファイルを用いている場合もあるかと思います。git-wt には --copyignored といったオプションがあり .env も worktree へコピーしてくることが可能です。

でも .env って大体シークレットが書かれていて、その環境下で Coding Agent さんを動かすのはちょっと不安です。十中八九大丈夫でしょう（個人の感想です）が、こういう心配事は脳のリソースを無駄に消費します。

そこでくふうカンパニーでは **dotenvx × 1Password CLI** の採用を進めています。この構成により、シークレットが必要になったタイミングで、開発者が Touch ID などによる認証を行い、明示的に許可した上でシークレットを取得できるようにしています。

-   **dotenvx** は .env の内容を暗号化し、Git 管理下に置きます。  
    シークレットを利用する際は、対応する秘密鍵で復号します。
    
-   **1Password CLI** は、1Password に保管されているシークレットを CLI 経由で安全に参照・管理できるようにします。
    

これらを組み合わせることで、**dotenvx の秘密鍵を 1Password に保管**し、必要時に認証を経て取得・復号する運用を実現しています。

以下サンプルです。

```
$ DOTENV_PRIVATE_KEY_DEVELOPMENT=op://path/to/dotenvx_private_key \
        op run -- dotenvx run -f .env.development -- go run main.go
```

## 終わりに

今回紹介したツールや仕組みが、そのままフィットしない、あるいは適用が難しいケースもあると思います。むしろ、そのようなケースの方が多いのかもしれません。

とはいえ、「このツールは使えない」で終わらせるのではなく、前提となっている開発環境の仕組み自体を見直し、アップデートしていく視点が重要だと考えています。  
今回の例ではコンテナや DB を新たに用意しましたが、実際にはその後に DDL を適用する必要がありますし、高速化のためにボリュームのスナップショットを流し込むなど、追加のくふうが求められる場面も出てくるでしょう。これらの対応は仕組みをアップデートしておくことで比較的容易になるはずです。

本記事が、同様の課題に向き合う方の参考になり、開発環境を見直すきっかけになれば幸いです。
